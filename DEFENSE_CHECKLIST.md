## 📝 预料的问题与应答 (5 大高频 Q&A)

### Q1: "为什么 EID 命中率还是只有 28%,这不是很低吗?"

**应答**:
1. 这是从 0% 到 28% 的巨大进步 (G0/G1/G2 都是 0%)
2. **但 28% 确实还有优化空间,根本原因是检索瓶颈**
   - 当前 Recall@5 = 0.388 (61% 概率 top-5 找不到 gold)
   - 业界基准是 0.45-0.55 (即使优化也难超 60%)
3. **这不是生成层 LoRA 的错误**
   - LoRA 忠实地引用了检索返回的 EID
   - 问题在检索算法,需要 Reranker 对比学习 (16-24 小时大工程)
4. **完整系统的防线**:
   - 格式合规率 76.0% (LoRA 的成就)
   - EID 命中率 28% (受检索限制)
   - 幻觉率 2.0% (三层防线联合)

---

### Q2: "B1/B3 为什么不修?偷懒吗?"

**应答**:
1. **不是偷懒,是诚实的工程决策**
2. **边际收益 vs 成本分析**:
   - B1 修复: 2-3 小时数据标注 + 30 min 重训 → +5-10 pp 格式
   - B3 修复: 3-4 小时重训 → +3-5 pp 答案完整性
3. **都不改变核心结论**: "RAG + LoRA 将幻觉从 18% 降至 2.0%"
4. **这是成熟项目的做法**:
   - 主动暴露低优先级 Bad Case
   - 给下一步工作者留下清晰的修复路线图
   - 信任 > 完美

---

### Q3: "这和其他 RAG 系统有什么不同?"

**应答**:
1. **三层防线架构**:
   - L1-L4 检索 (证据约束)
   - L5 LoRA 微调 (反幻觉学习)
   - L7 YAML 规则 (可审计保证)
2. **关键实证发现** (学术贡献):
   - **发现 1**: 格式学习必须微调,prompt 补不了
     - G2 最强 prompt 还是 0% 格式
     - G3 +LoRA 一跃 76.0% 格式
   - **发现 2**: RAG 只能砍一半幻觉
     - G0→G1: 18% → 10% (-44%, RAG 的贡献)
     - G1→G3: 10% → 2% (-80%, LoRA 的贡献)
   - **发现 3**: <1B 中文小模型零样本无法学复杂格式
3. **这些实证规律对后续 RAG 系统有指导意义**

---

### Q4: "能部署到生产吗?成本怎么样?"

**应答**:
1. **可以,成本可控**
2. **资源消耗**:
   - 磁盘: 2.8 GB (模型 + 知识库)
   - 显存: 4.5 GB (RTX 2060 8GB 有富余)
   - 延迟: 10.5 秒 (vs 基线 9.8s,仅增加 7%)
3. **LoRA 轻量化优势**:
   - Adapter 只有 34 MB
   - 不需要修改原模型代码
   - 易于版本管理 + AB test
4. **吞吐量** (推理):
   - 单卡 8GB: ~每分钟 5-6 条查询
   - 足以支持企业级合规查询
5. **部署方式**:
   - 方式 A: 命令行脚本 (开发)
   - 方式 B: HTTP API (生产)
   - 方式 C: Colab T4/P100 (演示)

---

### Q5: "为什么选 Qwen2.5-0.5B?还有没有其他尝试?"

**应答**:
1. **为什么选 0.5B**:
   - <1B 模型在 8GB GPU 上可行
   - 推理快 (4-5 秒生成) + 成本低
   - 中文指令遵循能力足够
2. **其他尝试**:
   - 测试过 1.5B (吃显存 8.5GB,接近限制)
   - 测试过 MacBERT (作为辅线对标)
   - 最终选 0.5B 是平衡效果 + 部署的最优点
3. **微调参数**:
   - LoRA r=16, α=32 (轻量但有效)
   - 4-bit NF4 量化 (节省显存 50%)
   - 2 epoch 收敛 (42 分钟完成)
