## 3. 意图体系重新设计（解决"你好"问题）

### 3.1 旧体系的问题

现有 4 类：`case_retrieval / law_grounding / sanction_recommendation / trend_analysis`。

- 用户输入"你好" → 走 topic_guard → 命中"短 query 不强制领域词"通过 → 进入检索 → 拿一堆垃圾案例 → 生成莫名其妙答案。
- 用户输入"谢谢" → 被当成 case_retrieval。
- 用户输入"今天天气怎么样" → 被 topic_guard 拦截（✅）。

### 3.2 新体系（7 类 + 对应动作）

| 意图 | 示例 | 系统动作 |
|------|------|---------|
| `greeting` | 你好 / hi / 在吗 | 返回固定问候："你好，我是证券违规案例检索助手，可以帮你查历史案例、法规依据、处罚推荐和趋势分析" |
| `chitchat` | 你叫什么名字 / 你会做什么 | 返回能力介绍 |
| `out_of_scope` | 明天天气 / 写首诗 | 拒答："该问题不在证监会处罚案例范围内，请问是否需要帮你查询证券违规相关内容？" |
| `case_retrieval` | 哪些公司被罚过内幕交易 | 走 RAG 流程（事件级） |
| `law_grounding` | 内幕交易违反哪些法条 | 走 RAG 流程（事件+法条） |
| `sanction_recommendation` | 这段违规行为通常怎么处罚 | 走 RAG + 调用 MacBERT 分类辅线 |
| `trend_analysis` | 近五年操纵市场案件在增加吗 | 走 RAG + 统计聚合 |

### 3.3 如何训

- 每类写 **30 条种子样本**（5 个人每人 6 条，1 晚搞定）
- 用 **LLM 批量改写扩增到 500 条/类**（可用 GPT-4 或 DeepSeek 免费额度）
- 最终 3500 条，训 TF-IDF + LR，holdout 预期 F1 ≥ 0.92

## 4. Query 改写层设计（新加）

### 4.1 为什么必须做

**BM25 对词形极度敏感**。测试样例：

- 用户说"**抄股**内幕"→ BM25 找不到（知识库里都是"炒股"或"股票交易"）
- 用户说"**康美**案那种"→ 没有 Query 改写，BM25 把它当普通词查
- 用户第二轮问"**那它的法条是什么**"→ "它" 指谁？单独一句话无法检索

### 4.2 三个子模块

#### (a) 多轮共指消解（基于规则 + LLM fallback）

输入：
```
history = [
  {user: "康美药业被罚过什么", assistant: "...康美药业(600518)于2020年被证监会..."},
  {user: "那它的法条呢"}
]
```

输出改写后的 canonical_query：
```
"康美药业 处罚 违反的法条"
```

实现：**本轮问题含代词 → 从 history 最后一轮 assistant 答案中抽取 entity → 拼接**。
复杂情况（多代词、隔轮指代）走 LLM few-shot。

#### (b) 领域同义词扩展

`configs/synonyms.json`：

```json
{
  "内幕交易": ["内幕信息交易", "利用未公开信息交易", "老鼠仓"],
  "信息披露": ["信披", "定期报告披露", "公告披露"],
  "市场操纵": ["操纵市场", "操纵证券市场", "坐庄"],
  "炒股": ["股票交易", "买卖股票", "炒股", "抄股"],
  "上市公司": ["上市公司", "A股公司", "挂牌公司"],
  "罚款": ["罚款", "处以罚款", "被罚"]
}
```

检索时：query 先分词 → 每个词查同义词表 → BM25 用 OR 查询合并。

#### (c) 槽位抽取（正则 + 少量 LLM）

可抽取字段：
- **年份**：`(\d{4})年` / `近(\d+)年`
- **公司**：用 jieba 自定义词典 + 股票代码正则
- **违规类型**：预定义 6 类关键词匹配
- **机构**：证监会/各地证监局/交易所

抽到的槽位用作 L3a 的**元数据硬过滤**条件，大幅提升 top-k 精度。

## 5. 统一数据处理流水线（新加）

### 5.1 原始数据 → 四份衍生数据集

```
证监会处罚信息表.xlsx (14,740 行)
         │
         ├─→ event_corpus.jsonl (4,233 事件级文档)  ← 已有
         │        │
         │        ├─→ event_chunks.jsonl (29,314 chunks, 给检索)  ← 已有
         │        └─→ rag_qa_train.jsonl (5360 条 LoRA 训练)  ← 要造
         │
         ├─→ party_samples.jsonl (14,740 当事人)  ← 已有
         │        └─→ MacBERT 分类训练集
         │
         ├─→ intent_train.jsonl (3500 条 7 类 × 500)  ← 要造
         │
         └─→ eval_gold.jsonl (50 条人工标注)  ← 要造
```

### 5.2 切分原则（防泄漏）

所有数据集**统一按 EventID 切分**，而且**时间切分**：

| 子集 | 时间范围 | 占比 |
|------|---------|------|
| Train | 1994-2021 | ~80% |
| Val | 2022-2023 | ~10% |
| Test | 2024-2025 | ~10% |

**同一个 EventID 不会出现在不同子集**。统一切分脚本：`scripts/split_by_event.py`。

### 5.3 数据质量清单

- ❌ `PunishmentMeasure` 不进 L5 和辅线分类输入（会泄漏标签）
- ❌ 长度超过 20000 字符的 Activity 要截断（极端长文降低训练稳定性）
- ❌ 全为空字符的事件直接丢弃
- ✅ PunishmentType 为空的事件可以进 RAG 训练数据（当证据），但不能进分类训练集

## 6. LoRA 训练数据构造（重新规划）

针对你的担心"模板派生会不会让模型过拟合到模板句式"——我加入了**改写增强**和**负样本**两个手段。

### 6.1 数据分布（总 5360 条）

| 类别 | 数量 | 构造方式 | 防过拟合策略 |
|------|------|---------|------------|
| A. 案例检索 QA | 1500 | 模板派生 | 每条问题用 3 个不同问法 |
| B. 法条依据 QA | 1200 | 模板派生 | 同上 |
| C. 处罚推荐 QA | 1200 | 模板派生 | 同上 |
| D. 趋势统计 QA | 600 | 从聚合数据自动生成 | 年度/机构轮换 |
| E. 拒答样本 | 300 | 手写 + LLM 扩增 | 覆盖各类越界情形 |
| F. 问候/闲聊 | 200 | 手写 + LLM 扩增 | — |
| G. 多轮跟进 | 300 | 手工设计 2 轮对话 | 强制带代词消解样例 |
| H. 反幻觉负样本 | 200 | 故意设问"虚构公司 XX 的处罚"，答案必须是"未检索到" | 防编造 |

### 6.2 Prompt 模板（所有样本统一格式）

```
<|im_start|>system
你是证监会处罚案例检索助手。你的回答必须：
1. 严格基于下方"检索证据"，不得编造案例或法条
2. 每引用一个案例必须标注 [EventID=xxx]
3. 每引用一条法规必须标注 [法条：《xx法》第xx条]
4. 若证据不足，必须明确说"未检索到充分证据"
<|im_end|>
<|im_start|>user
【用户问题】{query}

【检索证据】
{evidence_block}
<|im_end|>
<|im_start|>assistant
{target_answer}
<|im_end|>
```

### 6.3 问题增强（让模型见到多种问法）

对每个原始事件，生成 3 种问法的 QA：

- 正式问法："涉及内幕交易的上市公司高管处罚案例有哪些"
- 口语问法："我司高管好像搞了内幕交易，会怎么处罚"
- 短问法："内幕交易处罚"

用 prompt 让 LLM（或手工）改写。

## 11. 需要你最终拍板的问题（在动手前回答）

### Q1. 意图是否扩到 7 类？

我强烈建议扩，否则"你好"这种输入就是 bad case。

### Q2. Query 改写层是否一定做？

做比不做多 1-1.5 天工作量，但对检索 Recall 帮助巨大。建议做，但**不训模型，只写规则 + 同义词表**。

### Q3. 槽位抽取走正则还是 NER 模型？

正则能覆盖 80%（公司名/年份/股票代码），剩余 20% 走 LLM few-shot。**不训 NER**。

### Q4. LoRA 训练是在本机跑，还是 Colab？

本机如果有 ≥8GB 显存的 N 卡，跑得动。否则必须上 Colab T4。
