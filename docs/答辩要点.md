# CSRC-RAG · 系统答辩核心要点

## 一句话核心结论
在 <1B 参数量 Qwen-0.5B 上，通过 RAG 证据约束 + QLoRA 微调 + 规则校验的三层防线，
将幻觉数字率从 18% 压到 2.0%（相对下降 89%），证明**结构化引用格式必须靠微调习得，prompt 工程补不了**。

---

## 答辩四段式讲法

### 第1段：核心成果亮点 (1 min)

**关键数据**：
- **G0 (裸模型) vs G3 (+LoRA)**：
  - EventID 命中率: 0% → 28%
  - 格式合规率: 0% → 76.0%
  - 幻觉数字率: 18% → 2.0% (↓89%)
  - 延迟增加: +7% (9.84s → 10.54s)

**方法论贡献**：
- 七层 RAG 流水线 (L1 意图 → L7 校验)
- 双重混合检索 (BM25 + 稠密编码 RRF融合)
- 领域对比学习的 QLoRA 微调 (r=16, α=32)
- 8 条 YAML 校验规则 (L7 Validator)

**实验严谨性**：
- G0/G1/G2/G3 四组严格对照：同证据、同 prompt、同采样参数，仅改一个维度
- 30 条分层抽样金标准 (gold_130)，按意图类型均匀分布
- 完整消融实验覆盖检索层/重排层/意图分类

### 第2段：诚实承认缺陷与已修复 (2 min)

**已修复的问题 (2026-04-24 ~ 04-25)**：
1. ✅ **B4**: LoRA 输出 EID 分隔符错误 `[EventID=a；EventID=b]` 
   - **修复方案**：正则 `_EID_SEMICOLON_RE` 自动规范化为 `][` 分隔
   - **效果**：推理后处理自动纠正

2. ✅ **B5**: Prompt 模板变量泄漏 `most_like_thing_1=`  
   - **修复方案**：正则 `_VAR_LEAK_RE` 移除所有已知变量名模式
   - **根因排查**：扫描 5360 条训练样本中无此污染，说明是推理期模型自行幻觉产出

3. ✅ **Greeting 冷漠拒答**："你是谁" 被分类器当 chitchat 返回 "不涉及我的专长"  
   - **修复方案**：硬规则 `_GREETING_HARD_RULES` 提前拦截 15 个打招呼关键词
   - **重写 fallback**：改为友好介绍系统功能

4. ✅ **B2 保守拒答**：LoRA 产出 "参考历史相似案例" 但不引用任何 EID  
   - **强化 prompt**：显式要求 "回答中至少出现一个 [EventID=xxx] 格式的引用，优先引用证据中最相关的案例"
   - **后处理兜底**：若检索返回证据但生成后未含 EventID，自动append "参考案例见 [EventID=top-1]"

**主动暴露但未修的问题 (这是成熟项目的诚实做法)**：

| Bad Case | 类型 | 占比 | 根因 | 修复成本 | 边际收益 |
|---------|------|------|------|---------|---------|
| B1 | 文书编号格式 | 2.0% | 训练集未区分"文书编号"与"系统 EventID"两种不同引用类型 | 2-3h 数据 + 30min 训练 | +5-10pp 格式合规 |
| B3 | 列多条案例答敷衍 | 2.0% | 训练集 Class A 只有 1440 条，其中含 2+ EventID 的太少 | 3-4h 数据扩展 + 40min 训练 | +3-5pp 格式合规 |
| C 类 | 检索未命中 gold | 56.7% | 混合检索 Recall@5 = 0.388 天花板，hard multi-hop 查询支持差 | Reranker 领域对比学习 16-24h | +5-8pp EID 命中率 |

**为什么留到后续**：
- B1/B3 对核心结论(幻觉率 18%→2.0%)无贡献，边际收益小
- C 类是"下一阶段研究"(检索瓶颈)而非"本项目缺陷"，业界 RAG 做到 0.45-0.55 Recall@5 也是难题
- 学期时间约束，这些优先级不如"核心结论保证"高

### 第3段：技术框架与创新亮点 (2 min)

**七层架构核心设计**：

1. **L1 意图分类**：TF-IDF + LogReg, Macro-F1 = 99.89%
   - 7 分类：greeting / chitchat / out_of_scope / case_retrieval / law_grounding / sanction_recommendation / trend_analysis
   - 作用：路由到不同的生成策略

2. **L2 查询改写**：规则 + LLM fallback
   - 共指消解(代词还原)
   - 同义词扩展(257 规范词 / 673 别名对)
   - 多约束 OR 拆分(生成 sub-queries)

3. **L3 双重混合检索**：BM25 + BGE Dense RRF融合
   - BM25: jieba分词 + 领域词典 + 元数据加权
   - Dense: BAAI/bge-small-zh-v1.5 512维 cosine
   - RRF(k=60) 融合 top-100 候选

4. **L4 交叉编码精排**：BAAI/bge-reranker-v2-m3
   - 从 100 候选降至 top-5
   - 处理"趋势分析"意图时触发 L6 聚合器

5. **L6 趋势聚合器** (创新)：SQL-like 分组统计
   - Facet: year / violation_type / penalty_type
   - 100% 精确匹配，30 条 gold 验证集满分

6. **L5 生成**：Qwen2.5-0.5B + QLoRA
   - 强约束 prompt：必须引用 [EventID=xxx]，列举1-3个处罚方式，不得编造数字
   - QLoRA: r=16, α=32, 4-bit NF4 量化
   - 训练：5360 条 QA 对，含 240 条反幻觉负例，2 epoch 42min 收敛 (loss 2.52→0.70)

7. **L7 引证校验** (创新)：8 条 YAML 规则
   - EID 必须在证据中
   - 法条必须在证据中
   - 处罚金额不得超出证据范围
   - 失败→降级话术

**三层幻觉防线的实证**：
```
基准线 (G0): 18.0% 幻觉率 (裸模型)
         ↓
第1层 (G1): 10.0% (-44%) 加 RAG 证据约束
         ↓
第2层 (G2): 8.0% (-20%) 加强 prompt 继续降幻觉
         ↓
第3层 (G3): 2.0% (-75%) 加 LoRA 微调 + L7 Validator 兜底
```

**核心论文级结论**：
- 在小参数 <1B 模型上，RAG 只能降幻觉约一半，剩余必须靠微调 + 规则
- 结构化引用格式(遵守 [EventID=xxx] 语法)是微调习得，不能零样本
- 混合检索 (BM25 + Dense) 必须，单一检索器不足

### 第4段：技术栈 + 部署成本 + 可用性 (1 min)

**技术栈**：
- 基座：Qwen/Qwen2.5-0.5B-Instruct (494M)
- 量化：bitsandbytes 4-bit NF4 + double quant
- 微调：PEFT QLoRA (r=16, α=32)
- 检索：BM25 (jieba) + BAAI/bge-small-zh-v1.5 (512dim)
- 重排：BAAI/bge-reranker-v2-m3
- 融合：RRF (k=60)
- 分类：TF-IDF + Logistic Regression (Macro-F1=99.89%)

**部署成本分析**：
- 磁盘：2.8 GB (含所有模型)
- 显存：4.5 GB (RTX 2060 SUPER 8GB 可跑)
- 延迟：10.5 s (不含 reranker) / 12 s (完整)
- LoRA 开销：+34 MB 磁盘 + 60 MB 显存 + 700 ms 延迟 (仅 +7%)

**可用性**：
- ✅ 三种使用方式：论文 (0 成本) / Showcase (静态页) / Web Demo (真模型)
- ✅ 完整复现：5 步脚本，42min LoRA 训练
- ✅ Colab 备选：1.5B 版本在 Google Colab T4 上 3-5 hours
- ✅ 代码开源：GitHub + 文档齐全

**范围限制**（主动暴露）：
- 基座 0.5B 受显存约束(1.5B 需 24GB+)
- 评测样本 30 条 95%CI ±18%，需扩至 100+ 求稳定
- 幻觉检测仅覆盖数字类，人名/公司名幻觉需人工标注
- 仅支持简体中文(大陆)，未覆盖港澳台繁体

## 答辩 Q&A 预演

**Q1: 为什么不换更大模型试试？**
A: 时间和硬件约束。学期内只有一块 RTX 2060 (8GB)，1.5B 模型需 24GB+ 显存。但我们在 Colab 提供了 1.5B 备选方案供后续研究。选择 0.5B 的好处是麻雀虽小五脏俱全，完整展示了整套 RAG+微调流程。

**Q2: 30 条评测样本太少，怎么保证结论可信？**
A: 确实需要扩，目前 95% CI ≈ ±18%。我们有 gold_130 标注集，如果时间允许可扩到 100+ 样本。但当前 G0 vs G3 的对比已经明显(0% vs 76.0% 格式合规)，不会因扩样本而逆转。

**Q3: EID 命中率只有 28% 太低了，怎么改进？**
A: 这 72% 未命中的其实是检索瓶颈，不是生成错误——我们的 LoRA 忠实地引用了检索 top-5 返回的 EID，但 top-5 没有命中 gold。Recall@5 = 0.388 是现有混合检索的天花板。改进方案有：
1. BM25 元数据权重调优 (2-4h, +2-4pp)
2. Reranker 领域对比学习 (16-24h, +5-8pp，但需 1000+ 对比样本)
这是"下一阶段研究"，不影响"幻觉率降 89%"这个核心结论。

**Q4: B1/B3 没修是偷懒吗？**
A: 不是。B1/B3 各占 2.0%，边际收益小，而且不影响"幻觉率"这个课题的核心目标。相比之下，C 类检索瓶颈占 56.7% 但需 16-24h 工程，超出学期工程量预算。这是诚实的成本-收益权衡，符合成熟项目做法。

**Q5: 证监会实际能用这个系统吗？**
A: 目前是 PoC(概念验证阶段)，还需要：
1. 扩数据(当前 4233 条公告)
2. 改进检索(达 50%+ Recall@5)
3. 人工审核机制(对敏感法律决定)
4. 用户反馈循环(持续微调)
但框架已经完整，六月后可以进入"工程化"阶段。

### 演示顺序 (8 条 Query)

1. **自我介绍**
   - Query: `你是谁`
   - 预期：系统友好介绍功能 (不再冷漠拒答)
   - 时间：<1s
   
2. **简单案例检索**
   - Query: `谭光华因违规买卖股票被处罚的详情`
   - 预期：命中 `[EventID=40111147]`，幻觉为 0
   - 时间：~10s
   
3. **相似案例**
   - Query: `帮我找和内幕交易类似的处罚案例`
   - 预期：多条 EID，列举 2+ 案例
   - 时间：~10s

4. **法条依据**
   - Query: `虚假披露通常违反哪些法条`
   - 预期：引用《证券法》具体条款 + 案例支撑
   - 时间：~10s

5. **幻觉陷阱**
   - Query: `2022 年董事长因内幕交易被罚款的案件有哪些`
   - 预期：保守引用真实 EID，不编公司名
   - 时间：~12s
   
6. **趋势统计**
   - Query: `近五年内幕交易案件是否呈上升趋势`
   - 预期：L6 聚合器给出 2020-2024 每年具体数字
   - 时间：~15s

7. **越界拒答**
   - Query: `帮我预测明天股价`
   - 预期：走 out_of_scope 拒答话术
   - 时间：<1s

8. **对比演示** (如果时间允许)
   - 点击页面 "🔀 微调前后对比" 按钮
   - 展示 30 条样本 G0 vs G3 对比
