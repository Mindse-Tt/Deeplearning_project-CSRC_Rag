# 证监会处罚案例 RAG 课程项目总体框架

## 1. 项目定位

### 1.1 推荐题目

基于 RAG 与微调的证监会处罚案例智能检索、处罚推荐与法规依据解释系统

### 1.2 对应课程赛道

- 赛道：`B` 大模型微调与应用（Standard Track）
- 研究类型：`独立研究`

### 1.3 为什么这样定位

课程要求中，赛道 B 明确允许并鼓励：

- 基于预训练大模型构建 `RAG`
- 使用 `LoRA / P-Tuning` 等方式进行微调
- 展示微调前后效果对比
- 缓解大模型幻觉与场景不适配问题
- 讨论部署成本、推理延迟与隐私合规

本项目的数据特征与课程要求天然匹配：

- 数据主体是中文长文本案例，适合做检索增强与问答
- `Activity（违规行为）` 文本信息量很高，适合做主证据字段
- `PunishmentType（处罚方式）` 可作为监督学习标签
- 数据既可支持 RAG，又可支持微调分类
- 最终可落成论文 + 代码 + 演示系统

## 2. 需求合同

### 2.1 Goal

构建一个面向证监会处罚案例的垂直 RAG 系统，能够围绕真实案情：

- 检索相似历史处罚案例
- 给出法规依据
- 推荐处罚类型
- 生成基于证据的解释性回答

### 2.2 Inputs

- 主数据：[证监会处罚信息表.xlsx](证监会处罚信息表.xlsx)
- 核心字段：
  - `M列 Activity / 违规行为`
  - `N列 Law / 违反的法律法规`
  - `W列 PunishmentType / 处罚方式`
  - `X列 SumPenalty / 处罚金额`
  - 以及 `DeclareDate`、`Promulgator`、`Party`、`Position`、`Relationship`

### 2.3 Constraints

- 必须符合课程赛道 B 的要求
- 需要有 RAG 主线
- 需要有微调或监督学习主线
- 需要有 baseline 对比
- 需要有幻觉控制与部署分析
- 需要能最终写成论文并支持答辩演示

### 2.4 Deliverable

- 一篇 8-12 页学术风格研究论文
- 一个可复现代码仓库
- 一个可演示的端到端系统原型
- 一套完整实验结果与对比分析

### 2.5 Verification

- 检索层指标达到可解释、可比较的效果
- 处罚类型预测在合理 baseline 上取得稳定提升
- RAG 回答具有证据引用与法规依据
- 系统可在答辩中完成真实输入到结果输出的闭环演示

## 3. 数据证据与建模依据

以下证据来自对当前 Excel 数据的本地解析结果。

### 3.1 数据规模

- 实际字段数：`24`
- 数据行数：`14740`
- 唯一事件数：`4233`
- 多行事件数：`2109`
- 年份范围：`1994-2025`

### 3.2 为什么不能把整张表直接当成普通样本表

这份数据同时混合了两种粒度：

- 事件级：一个处罚事件对应一段完整案情与法规描述
- 当事人级：同一事件下可能有多个被处罚主体

如果不先区分粒度，直接按行训练和评估，会出现两个问题：

- 同一个 `EventID` 可能同时出现在训练集和测试集，导致泄漏
- 事件级文本与当事人级标签混在一起，任务定义失真

### 3.3 为什么 `M列 Activity` 是核心字段

`Activity` 的文本长度和内容信息最丰富：

- 非空：`14740`
- 中位长度约：`1503` 字符
- `p90` 长度约：`4170` 字符

这意味着它不仅适合：

- 做语义检索
- 做案例摘要
- 做处罚类型预测输入
- 做法规依据生成

而且能成为系统中最强的“主证据文本”。

### 3.4 为什么 `W列 PunishmentType` 适合作为主标签

`PunishmentType` 的标签结构较清楚，且覆盖率高：

- 非空：`14740`
- 唯一标签组合数：`45`
- 多标签分布：
  - 1 个标签：`2405`
  - 2 个标签：`10231`
  - 3 个标签：`1998`
  - 4 个标签：`105`
  - 5 个标签：`1`

这说明该任务天然是一个多标签分类任务，适合作为课程项目中的监督学习主任务。

### 3.5 为什么不能把 `PunishmentMeasure` 放进输入

`PunishmentMeasure` 会直接写出最终处罚决定，例如“给予警告并罚款多少元”等。如果将其作为模型输入，再去预测 `PunishmentType` 或 `SumPenalty`，会造成严重的标签泄漏，实验结果失真，不具有研究价值。

## 4. 项目大方向与边界

### 4.1 推荐的大方向

项目不做成“开放聊天机器人”，而做成“证据驱动的监管处罚案例分析系统”。

系统核心能力定义为：

1. 案例检索
2. 法规依据问答
3. 处罚类型推荐
4. 基于证据的解释生成

### 4.2 本项目建议回答的问题

#### 类型 A：案例检索问题

- 这段违规行为最像哪些历史处罚案例？
- 同类案件通常由哪些机构处罚？
- 某类违规行为近几年最典型的处罚案例有哪些？

#### 类型 B：法规依据问题

- 该行为通常违反哪些法条？
- 过去相似案例主要援引了哪些法律法规？
- 不同年份对同类行为适用的法规是否有差异？

#### 类型 C：处罚推荐问题

- 根据案情描述，更可能对应哪些处罚方式？
- 是否可能出现市场禁入？
- 对上市公司高管与普通员工，处罚类型是否存在差异？

#### 类型 D：统计分析问题

- 某类违规行为最常见的处罚方式是什么？
- 近五年某类案件的处罚是否趋严？
- 不同监管机构的处罚风格有何差异？

### 4.3 必须收住的边界

本项目应当明确不做以下内容作为主线：

- 不做泛化开放域问答
- 不做完整处罚文书自动生成
- 不做精确罚款金额回归作为主任务
- 不做脱离证据的自由推理型聊天

原因：

- 这些方向更容易幻觉
- 更难形成稳定量化指标
- 更不适合课程项目在 10 周内完成
- 与当前数据最强信号不完全匹配

## 5. 技术路线

整体采用“双主线 + 一个演示系统”的结构。

### 5.1 主线一：事件级 RAG 检索问答

目标：

- 围绕事件级案情文本，建立可检索知识库
- 支持相似案例检索、法规依据问答和解释生成

基本流程：

1. 以 `EventID` 聚合事件级文档
2. 以 `Activity` 为主文本，补充 `Law`、`FileName`、`Promulgator`、`DeclareDate`
3. 做文档清洗、切块、向量化与索引
4. 在线查询时进行混合召回
5. 使用 reranker 做精排
6. 将 top-k 证据片段送入生成模型
7. 输出“结论 + 引用案例 + 引用法条 + 说明”

### 5.2 主线二：当事人级处罚类型预测

目标：

- 根据案情和当事人上下文，预测 `PunishmentType`

输入建议：

- `Activity`
- `Law`
- `Party`
- `Position`
- `Relationship`
- `IsListedCom`
- `Promulgator`
- 时间特征

标签：

- `PunishmentType` 多标签分类

可选扩展：

- `SumPenalty` 金额分桶预测
- `ViolationType` 辅助任务学习

### 5.3 演示系统

目标：

- 让老师在答辩中看到完整闭环，而不是只看离线实验表格

建议功能页：

- 智能问答页
- 相似案例检索页
- 处罚推荐页
- 统计分析页

## 6. 检索策略设计

### 6.1 为什么不能只用单一路径检索

监管处罚文本有两个明显特点：

- 术语固定、法条固定，词法匹配很重要
- 表述方式多样，语义匹配同样重要

因此单用 BM25 或单用向量检索都不稳，推荐使用混合检索。

### 6.2 推荐检索流程

#### 第一步：预过滤

按元数据做轻量过滤：

- 年份范围
- 是否上市公司
- 发布机构
- 处罚类型

#### 第二步：双路召回

- 路径 A：BM25 词法检索
- 路径 B：Dense Embedding 语义检索

#### 第三步：结果融合

推荐：

- `RRF` 融合
或
- 加权分数融合

#### 第四步：重排

对候选文档使用 cross-encoder reranker 进行精排。

#### 第五步：RAG 生成

将 top-k 片段输入生成模型，限制回答必须基于检索证据输出。

### 6.3 切块策略

优先尝试两种方案并比较：

- 方案 A：按字符切块，`300-500` 汉字，重叠 `50-100`
- 方案 B：按自然段或条目切块，再限制最大长度

建议保留两类索引：

- 事件级摘要索引
- 片段级详细索引

这样可以同时支持“找整案”和“找证据片段”。

## 7. 模型选择与对比方案

### 7.1 检索层模型

推荐优先候选：

- Embedding：
  - `BAAI/bge-m3`
  - `Qwen/Qwen3-Embedding-0.6B`
- Reranker：
  - `BAAI/bge-reranker-v2-m3`
  - `Qwen/Qwen3-Reranker-0.6B`

选择依据：

- `bge-m3` 支持长文本、稀疏/稠密/多向量检索，并且模型卡直接建议在 RAG 中使用“混合检索 + 重排”
- `Qwen3-Embedding-0.6B` 和 `Qwen3-Reranker-0.6B` 支持 100+ 语言、32K 上下文、指令感知，适合中文长文本检索场景

### 7.2 生成层模型

推荐优先候选：

- `Qwen/Qwen2.5-7B-Instruct`

选择依据：

- 中文能力较强
- 支持长上下文
- 指令遵循能力较好
- 适合作为 RAG 的生成模型

### 7.3 微调分类层模型

推荐基线：

- `hfl/chinese-macbert-base`
- `hfl/chinese-roberta-wwm-ext`

选择依据：

- 中文文本分类常用基线
- 易训练、易复现
- 适合做处罚类型多标签分类

### 7.4 必做 baseline

检索层 baseline：

- `BM25 only`
- `Dense only`
- `Hybrid`
- `Hybrid + Reranker`

生成层 baseline：

- `LLM 直接回答（无检索）`
- `RAG + 通用生成模型`

预测层 baseline：

- `Logistic Regression / TF-IDF`
- `Chinese RoBERTa / MacBERT`
- `微调后的改进模型`

## 8. 验证与评估方案

### 8.1 数据切分原则

必须至少满足以下两条：

- 按 `EventID` 切分，避免同一事件泄漏
- 优先使用时间切分，模拟真实部署场景

推荐：

- 训练集：较早年份
- 验证集：中间年份
- 测试集：最近年份

### 8.2 检索层指标

- `Recall@k`
- `MRR`
- `nDCG@10`

评估逻辑：

- 查询一个案件描述
- 看系统是否能召回同类处罚案例
- 看正确案例是否出现在前几位

### 8.3 预测层指标

由于 `PunishmentType` 是多标签任务，建议使用：

- `Micro-F1`
- `Macro-F1`
- `Hamming Loss`
- `Subset Accuracy`

### 8.4 RAG 回答质量评估

建议采用“自动指标 + 人工评估”结合：

- 自动指标：
  - 引用证据命中率
  - 法条引用正确率
  - 回答中是否包含检索来源
- 人工评估：
  - 回答是否基于证据
  - 回答是否逻辑一致
  - 回答是否出现幻觉
  - 回答是否具有业务解释价值

### 8.5 幻觉缓解与量化

课程要求明确提到幻觉问题，因此必须单独设计。

建议比较：

- 无 RAG
- 有 RAG，但无来源约束
- 有 RAG，且强制引用证据

重点观察：

- 是否捏造法条
- 是否捏造历史案例
- 是否给出超出证据范围的结论

## 9. 消融实验建议

至少做以下消融：

1. `BM25 vs Dense vs Hybrid`
2. `是否加入 Reranker`
3. `不同 chunk 大小`
4. `是否使用 metadata filter`
5. `无 RAG vs 有 RAG`
6. `无微调 vs 微调`
7. `仅 Activity vs Activity + Law + Party/Position`

这些消融足以支撑论文中“方法有效性”的论证。

## 10. 部署与演示建议

### 10.1 推荐工程结构

- 后端：`FastAPI`
- 检索索引：`FAISS + BM25`
- 前端：`Streamlit` 或 `React/Vue`

如果时间紧：

- 先做 `Streamlit`

如果要更像正式系统：

- 前端：`React`
- 后端：`FastAPI`

### 10.2 建议接口

- `/search`：相似案例检索
- `/qa`：RAG 问答
- `/predict`：处罚类型推荐
- `/stats`：统计分析
- `/case/{event_id}`：单案例详情

### 10.3 答辩演示流程

建议答辩现场演示一个真实输入：

1. 输入一段违规行为描述
2. 系统召回相似案例
3. 展示匹配法规
4. 给出处罚方式推荐
5. 展示生成解释

这样可以把检索、预测和生成三层价值一次性讲清楚。

## 11. 为什么这个方案有依据

### 11.1 依据一：课程要求

该方案完全贴合课程对赛道 B 的要求：

- 使用 RAG
- 可加入微调
- 可展示微调前后对比
- 可量化幻觉缓解
- 可讨论部署成本与时延
- 可形成论文 + 代码 + Demo

### 11.2 依据二：数据结构

本地数据分析已经表明：

- `Activity` 文本足够长且信息密度高
- `PunishmentType` 覆盖率高且适合多标签学习
- 事件级与当事人级并存，天然适合“双主线建模”

### 11.3 依据三：模型文档

公开模型卡支持上述技术选择：

- `bge-m3` 模型卡明确推荐在 RAG 中使用“混合检索 + 重排”
- `Qwen3-Embedding / Qwen3-Reranker` 支持长文本、多语言和指令感知
- `Qwen2.5-7B-Instruct` 适合长上下文生成与结构化输出
- `MacBERT` 适合作为中文分类基线

## 12. 当前建议的项目结论

如果以课程评分、数据特征、10 周项目周期和答辩展示效果综合判断，当前最稳妥且上限足够高的项目边界是：

### 主任务

基于 `Activity` 的证监会处罚案例 RAG 检索与法规依据问答

### 辅任务

基于案情和主体信息的 `PunishmentType` 多标签预测

### 展示形态

一个可交互网页 Demo，展示：

- 输入案情
- 检索相似案例
- 引用法规依据
- 推荐处罚类型
- 输出解释性回答

### 不建议当前作为主线的内容

- 精确罚款金额预测
- 自动生成完整处罚决定书
- 完全开放域聊天
