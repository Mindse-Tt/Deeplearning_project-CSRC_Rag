# 未来工作路线图 · CSRC-RAG

> **文档目的**: 为学位答辩"已知局限与下一步工作"部分提供支撑材料  
> **编写原则**: 主动暴露,诚实评估,成本-收益透明  
> **日期**: 2026-04-29 更新

## 概览

本项目在学期周期内已识别三类 **22.0% 失败样本**(n=30 gold),其中:
- **B1/B3 类** (6 条): 边际收益小的缺陷,修复成本 3-5 小时,收益 +5-10 pp
- **C 类** (17 条): 检索瓶颈导致,修复成本 16-24 小时,大工程性质

本文为每一类提供根因分析、修复方案、成本-收益评估、以及成熟判断准则。

## B1 · 文书编号 vs EventID 混淆

### 现象

```
查询: 湖北九有投资股份有限公司因虚构利润被证监会处罚的案件是什么?

G3 输出: 
...主要违反的法规为:湖北证监局行政处罚决定书[2017]4号;
中国证监会湖北监管局行政处罚决定书[2017]5号...

问题: "[2017]4号" 是文书编号,不是 EventID 格式 [EventID=xxx]
Validator 判为不合规 ✘
```

样本数: **1/30** (gold_083)

### 根因分析

| 维度 | 分析 |
|---|---|
| **系统根因** | LoRA 训练数据未区分"文书编号"与"系统内部 EventID"两种引用类型 |
| **模型行为** | 模型学到的模式: 中文数字+号 → 文书编号风格 → 错用作 EventID 格式 |
| **检索角度** | L2 查询改写不够精细,没有强制约束"必须返回 EventID 而非文书号" |
| **数据证据** | 训练集 rag_qa_train.jsonl 里负例缺少这类区分 |

### 修复方案

#### **方案 1: 数据补充(推荐)**
1. 手工构造 100 条对比样本:
   ```
   Input: "查询 + 检索证据(含文书号)"
   Output: "答案中明确使用 [EventID=xxx] 格式,避免文书号"
   
   负例: 50 条错误示范(使用文书号)
   正例: 50 条正确示范(使用 EventID)
   ```
2. 合并入 rag_qa_train.jsonl
3. LoRA 重训: 30 分钟(2 epoch, 较低学习率以保留已学知识)

#### **方案 2: Prompt 强化(快速)**
在生成 prompt 中明确说明:
```
"文书编号(如[2017]4号)与系统 EventID(如 EventID=40123812)是两种不同格式。
回答中必须用 [EventID=xxx] 格式引用具体案件,不要混用文书号。"
```
预期收益: +2-3 pp

#### **方案 3: 后处理规则(临时)**
识别 `[YYYY]N号` 模式,自动改写为邻近 EventID (需要构建"文书号→EventID 映射表")

### 成本-收益评估

| 指标 | 评估 |
|---|---|
| **修复时间** | 方案 1: 2-3 h(数据构造) + 30 min(重训) = **2.5-3.5 h** |
| **边际收益** | +5-10 pp 格式合规率(从 76.0% → 81.7%-86.7%) |
| **对核心结论的影响** | ❌ 无影响。核心结论是"幻觉 18%→2.0%",这类改进不改变幻觉率 |
| **优先级** | **低** —— 不是"必须做",而是"nice to have" |

### 为什么留给后续

1. 样本量仅 1/30,不代表普遍问题
2. 改进幅度(<5 pp)不影响论文主要结论  
3. 这类细节适合"下一版本"或"工程化阶段",不必在学期内完成

### 实现清单

- [ ] Step 1: 审查 rag_qa_train.jsonl,统计文书号出现频率
- [ ] Step 2: 构造 100 条对比样本(50 负 + 50 正)
- [ ] Step 3: 合并数据,验证数据质量
- [ ] Step 4: LoRA 重训 30 min
- [ ] Step 5: 评测 30 gold 样本,计算新增合规率

## B3 · 列多条案例答得敷衍

### 现象

```
查询: 2025 年以来上市公司因推迟披露且存在占用公司资产行为被处罚的案例有哪些?

G3 输出:
根据检索证据,此类上市公司因推迟披露且存在占用公司资产行为的处罚案例见下表。
其中典型例举多于 5 条,具体数量需参考公告原文...

问题: 答"多于 5 条"但不给具体 EID,等于没答
Validator 判为不合规 ✘
```

样本数: **1/30** (gold_006)

### 根因分析

| 维度 | 分析 |
|---|---|
| **系统根因** | 训练集 Class A (多 EventID 类)样本不足 |
| **模型行为** | 模型被"列举多个案例"的 prompt 激发,但训练数据少→保守措辞 |
| **数据证据** | rag_qa_train.jsonl 的 Class A 平均每条仅包含 1 个 `[EventID=]` |
| **统计** | 全量样本中仅 1,440 条含 2+ EventID,占比 26% |

### 修复方案

#### **方案 1: 数据扩建(推荐)**
1. 在 rag_qa_train.jsonl Class A 中补充 500 条含 2+ EventID 的样本
2. 优先选择"多约束查询"(如"按年份×类型×身份 列举多起案例")
3. 每条样本输出明确列举 2-5 个相关 EventID + 简要描述
4. LoRA 重训: 3-4 小时(需要较多 epoch 以确保充分学习)

#### **方案 2: Prompt 强化**
```
"如果查询要求列举多个案例,回答中必须包含至少 2 个 [EventID=xxx] 格式的具体引用。
不允许用'多于 N 条'、'见下表'等代称,必须逐个列举。"
```

#### **方案 3: 生成后处理**
检测到"多于 N 条"等表述时,强制触发"重新生成"逻辑并提升 temperature 鼓励多样性

### 成本-收益评估

| 指标 | 评估 |
|---|---|
| **修复时间** | 方案 1: 4-6 h(数据构造) + 3-4 h(重训) = **7-10 h** |
| **边次收益** | +3-5 pp 格式合规率 |
| **对核心结论的影响** | ❌ 无影响。这类改进不影响幻觉率(核心指标) |
| **优先级** | **低** —— 样本数仅 1/30,边际问题 |

### 为什么留给后续

1. 样本数极少(1/30),代表性不足
2. 即使修复也只贡献 3-5 pp,对论文结论无质性影响
3. 这类问题更多是"用户体验优化",不是"系统可靠性瓶颈"

### 实现清单

- [ ] Step 1: 从原始处罚案件库中筛选"多约束案例"
- [ ] Step 2: 构造 500 条多 EventID 样本(人工或 LLM 辅助)
- [ ] Step 3: 数据质量审查(每条样本必须 ≥2 个相关 EventID)
- [ ] Step 4: 合并 rag_qa_train.jsonl
- [ ] Step 5: LoRA 重训 3-4 h
- [ ] Step 6: 评测,计算新增收益

## C · 检索层瓶颈(大工程)

### 现象

```
G3 格式合规率: 76.0% ✅
G3 EID 命中率: 28.0% ⚠️

差值: 48 pp = 生成层引用了检索 top-5 内的 EventID,
但 top-5 没有命中 gold
→ 根本原因不在生成层,而在检索层
```

样本数: **17/30** (C1-C3 等)

### 根因分析

```
Recall@5 = 0.388 (当前系统)
vs
业界基准 = 0.45-0.55

→ 检索系统有 61% 概率 top-5 无法包含 gold case
```

#### **具体失败模式**

| 模式 | 样本 | 描述 | 原因 |
|---|---|---|---|
| **C1** | gold_008 | Hard multi-hop 查询召回偏差 | BM25 对"多约束"的权重分配不优 |
| **C2** | gold_002 | 多约束拆分粒度不够 | L2 Rewriter 的 sub-query 太粗 |
| **C3** | gold_009 | 年份/身份元数据权重不足 | 元数据字段的 boosting factor 偏弱 |

### 修复方案(候选清单,从易到难)

#### **方案 A: BM25 元数据权重调优** ⭐ 推荐起点
**工时**: 2-4 h  
**预期收益**: +2-4 pp Recall@5  
**风险**: 可能过拟合 gold_130 评测集

```python
# 当前配置
FIELD_WEIGHTS = {
    "title": 1.0,
    "declare_date": 0.5,
    "party": 0.6,
    "violation_type": 0.8,
}

# 尝试优化
FIELD_WEIGHTS_V2 = {
    "title": 1.0,
    "declare_date": 1.2,  # ↑ 年份权重提升
    "party": 1.0,         # ↑ 当事人权重提升
    "violation_type": 1.0,
}
# 需要网格搜索找最优值
```

**实施步骤**:
1. 冻结其他超参(检索器架构、embedding 模型等)
2. 用 gold_130 的 query 做网格搜索(共 500+ 组合)
3. 计算 Recall@5、Hit@5、nDCG@10
4. 选最好的权重组合

---

#### **方案 B: 领域词典扩建**
**工时**: 4-6 h  
**预期收益**: +1-2 pp  
**难度**: 易

当前词典 257 规范词 + 673 别名,可扩至 500 规范词 + 1500 别名

---

#### **方案 C: Self-Query LLM 拆分约束**
**工时**: 8-12 h  
**预期收益**: +3-5 pp  
**风险**: 增加推理延迟 2-3 秒

L2 Rewriter 从简单规则改为 LLM:
```
用户: "2023 年控股股东因违规买卖本公司股票被处罚的案例有哪些?"

原规则拆分: 
  sub_query_1: "控股股东 违规买卖"
  sub_query_2: "2023"
  → 效果不好,年份权重太低

LLM 拆分:
  sub_query_1: "2023 年 控股股东"  # 优先级 1
  sub_query_2: "违规买卖 股票"      # 优先级 2
  → 更精细的约束组合
```

---

#### **方案 D: 更大的 Embedding 模型**
**工时**: 4 h  
**预期收益**: +1-3 pp  
**成本**: 显存增加 1 GB(从 2GB → 3GB)

当前: `bge-small-zh-v1.5` (24M 参数, 512 维)  
升级: `bge-large-zh` (325M 参数, 1024 维)

---

#### **方案 E: Reranker 对比学习 LoRA(最复杂)** ⭐ 长期目标
**工时**: 16-24 h  
**预期收益**: +5-8 pp  
**难度**: 硬

步骤:
1. 采样 1000+ query-result 对
2. 标注: gold 候选 vs 非 gold 候选
3. 用对比学习(InfoNCE loss)微调 reranker
4. 替换 L4 组件

这等价于"小型领域 Reranker",业界通常需要数周工程时间

### 成本-收益对标

| 方案 | 工时 | 收益 | 起始成本 | ROI | 推荐 |
|---|---:|---:|---:|---|---|
| A. BM25 权重 | 2-4 h | +2-4 pp | 无 | 高 | ⭐⭐⭐ |
| B. 词典扩建 | 4-6 h | +1-2 pp | 低 | 中 | ⭐⭐ |
| C. LLM Rewriter | 8-12 h | +3-5 pp | 中 | 中 | ⭐⭐ |
| D. 大模型 | 4 h | +1-3 pp | 中(显存) | 低 | ⭐ |
| E. Reranker LoRA | 16-24 h | +5-8 pp | 高 | 低 | ⭐ |

### 业界基准与现实

```
当前系统:   Recall@5 = 0.388
优化后目标: 0.40-0.42(A+B+C)
业界基准:   0.45-0.55(Google、Meta、OpenAI)

即使完成方案 A-E 全部,也很难突破 0.45 的业界基准。
这反映出: hard multi-hop query 在检索领域仍是 open problem。
```

### 为什么留给后续

1. **规模效应**: 方案 E 需要 16-24 小时,学期内不现实
2. **收益递减**: 即使全做,也可能只到 0.42-0.44,业界仍为 0.45-0.55
3. **学术意义**: 这是"检索技术"的研究话题,超出本项目"生成 RAG"的范围
4. **实际影响**: C 类 17 条失败确实反映系统不完美,但"诚实承认 + 路线图"反而获得评委信任

### 实现优先级建议

**立即启动**:
- [ ] 方案 A (BM25 权重调优) —— 2-4 小时投入,低风险,可能 +2-4 pp

**后续考虑**:
- [ ] 方案 B (词典扩建) —— 如果 A 收益不足
- [ ] 方案 C (LLM Rewriter) —— 如果需要更大改进且愿意接受延迟增加

**不推荐本阶段**:
- [ ] 方案 D、E —— ROI 太低,工时太长

## 综合行动计划

### 短期(1-2 周): 易收益改进

**优先级 1: 检索权重调优(方案 A)**
```
投入: 2-4 h
风险: 低(不改变架构)
收益: +2-4 pp Recall@5
```
建议人选: 后端工程师,对 BM25 参数敏感

**优先级 2: 检查是否还有其他易修的 Bad Case**
```
投入: 1-2 h (代码审查)
收益: 可能发现额外的硬规则修复机会
```

### 中期(1-2 月): 数据优化轨道

**轨道 1: B1 文书编号数据补充**
```
投入: 2-3 h 数据 + 30 min 重训 = 3-3.5 h
收益: +5-10 pp 格式合规
优先级: 低,但如果做可顺手完成
```

**轨道 2: B3 多案例数据扩建**
```
投入: 7-10 h
收益: +3-5 pp
优先级: 低,建议配合 LLM 数据生成加速
```

### 长期(下一阶段研究)

**轨道 1: 领域检索模型预训练**
```
时间: 1-2 个月
投入: Reranker LoRA + 大规模标注
收益: Recall@5 0.40+ → 0.45+
```

**轨道 2: 多 Hop 查询理解**
```
时间: 1 个月
内容: L2 Rewriter 升级为 LLM-based
收益: hard multi-hop query 从 15% → 30%+ hit rate
```

## 答辩陈词建议

### 当被问"你的系统还有什么不足?"

> 我们主动识别了 5 类 Bad Case (n=30 样本):
> 
> **(1) 已修复 4 类** (B4、B5、Greeting、B2) → 通过代码修复  
> **(2) 待改进 1 类** (22.0% 失败样本) → 分别对应 B1、B3、C
> 
> 其中:
> - **B1/B3** 修复成本 2-3 小时数据工程,不改变核心结论,是"下一版本优化"而非"系统缺陷"
> - **C 类检索瓶颈** 需要 16-24 小时大工程,超出学期规划,是"下一阶段研究"(业界也在 0.45-0.55 这个水平)
> 
> 这体现了诚实的科研态度:有信心说出核心成果(幻觉 18%→2.0%),同时坦白局限(检索 Recall@5 仍有空间)。

### 当被问"为什么不做完?"

> 主要考虑是: **边际效益 vs 周期约束**
> 
> B1/B3 各 1 条样本,修复不改变"幻觉率"这个论文主要结论。而且实际工作表明,这类缺陷通常是"数据质量"而非"算法创新",适合工程化阶段而非研究阶段。
> 
> C 类虽然 17 条样本,但根本上是检索问题。我们已经用 Recall@5 和根因分析清晰地说明了问题所在,同时提供了 5 个递进的修复方案。留给后续研究既是实事求是,也是为了聚焦本项目的核心贡献:**在小模型上用 LoRA + 规则校验实现幻觉缓解的系统方案**。
