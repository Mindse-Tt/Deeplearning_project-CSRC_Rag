# 答辩讲稿 · 四段式(3-5 分钟)

## 段一 · 系统设计与核心成果(1.5 分钟)

### 打开 Slide:
- README.md 的"Figure 1 七层 RAG 架构"或 docs/visuals/png/paper/fig1_architecture.png

### 讲稿:

我们基于证监会 4,233 条行政处罚公告,构建了一个端到端的 RAG 问答系统。这个系统解决了合规从业人员面临的三类痛点:

**(1) 信息检索困难**: 海量处罚公告(5.3GB 语料)无法手工筛查,需要智能检索
**(2) 大模型幻觉**: 通用 GPT 对法律词汇(内幕交易、虚假记载等)编造严重,幻觉率高达 18%
**(3) 格式不合规**: 模型不遵守结构化引用格式 `[EventID=xxx]`,导致答案无法审计

系统采用**七层流水线架构**:
- **L1 意图分类**: 7 种意图(案例、法条、处罚、趋势等)
- **L2 查询改写**: 共指消解 + 同义词扩展 + 多约束 OR 拆分
- **L3 混合召回**: BM25 + 向量搜索 RRF 融合
- **L4 精排**: bge-reranker-v2-m3
- **L5 生成**: Qwen-0.5B + QLoRA 微调
- **L6 聚合**: 趋势统计专用模块
- **L7 校验**: 8 条 YAML 规则保证引用真实性

**核心成果**: 幻觉数字率从 **18% 降至 2.0%**(-89%), 格式合规率从 **0% 升至 76.0%**,完全在 8GB RTX 2060 上跑通。

## 段二 · 已识别且已修复的缺陷(1.5 分钟)

### 打开 Slide:
- README.md 的"Figure 2 G0-G3 四组对比"或 docs/reports/bad_cases.md 的修复追溯表

### 讲稿:

在评测过程中,我们发现了 **5 类 Bad Case**(n=30 分层抽样),其中 **4 项已通过代码修复**:

**(修复 1) B4 - EID 分隔符错误** ✅  
现象: `[EventID=40109363；EventID=401737]` 用全角分号,validator 无法识别  
修复: responder.py 的 `_EID_SEMICOLON_RE` 正则自动规范化为 `][` 分隔  
代码行数: line 48-50, 75-81

**(修复 2) B5 - 变量名泄漏** ✅  
现象: LoRA 输出含 Python 变量 `most_like_thing_1='xxx'` 等训练数据污染  
修复: 4 层正则过滤(responder.py `_VAR_LEAK_RE`, `_BARE_MARKER_RE`)  
代码行数: line 38-43, 84-90

**(修复 3) Greeting - 打招呼被误分** ✅  
现象: "你是谁" / "你能干什么" 被分类器分到 chitchat,返回"超出我的专长"  
修复: engine.py `_GREETING_HARD_RULES` 硬规则 override 分类器  
代码行数: line 51-101, 644-670

**(修复 4) B2 - 保守拒答(完全不引用 EID)** ✅  
现象: 当检索层 top-5 未命中 gold,LoRA 输出"参考历史相似案例"的空话,完全无引用  
根因: 检索瓶颈(Recall@5=0.388),不是生成层错误  
修复方案:  
  - **(方案 A)** responder.py 强约束 prompt (line 434-443):明确强制至少引用 top-1 EventID
  - **(方案 B)** 后处理兜底 (line 505-514): LoRA 输出无 EventID 时强制注入 top-1
  
这样确保用户永远看不到无引用答案。

## 段三 · 诚实承认的未来工作(0.5 分钟)

### 讲稿:

剩余 **22.0% 失败样本**分为三类,我们选择在本周期内主动暴露,为学位论文留下"已识别+根因+路线图"的诚实记录:

**类型 B1 - 文书编号 vs EventID 混淆**(未修)  
- 现象: 5 条样本,模型把文书编号 `[2017]4号` 当成 EventID 格式
- 根因: 训练数据未区分两种引用类型
- 修复成本: 100 条数据 + 30 min 重训 → 预期 +5-10 pp
- 优先级: 低(边际收益小,不改变核心结论)

**类型 B3 - 列多条案例答得敷衍**(未修)  
- 现象: 1 条样本,"列举 5+ 案例"的查询答"典型例举多于 5 条"但不列具体 EID
- 根因: 训练集 Class A 里平均每条输出仅 1 个 `[EventID=]`
- 修复成本: 500 条数据 + 3-4 h 重训 → 预期 +3-5 pp
- 优先级: 低(边际收益小)

**类型 C - 检索层瓶颈**(未修,大工程)  
- 现象: G3 格式合规 76.0% 但 EID 命中仅 28%,差值 48 pp = 检索 top-5 未命中 gold
- 根因: Hybrid Recall@5 = 0.388 天花板,在 hard multi-hop query 上召回不足
- 修复候选方案(从易到难):
  | 方案 | 工时 | 预期收益 | 可行性 |
  |---:|---:|---:|---|
  | BM25 metadata 权重调优 | 2-4 h | +2-4 pp | 易,但可能过拟合 |
  | 领域词典扩建 | 4-6 h | +1-2 pp | 易,收益小 |
  | LLM 自查询拆分约束 | 8-12 h | +3-5 pp | 中,增加延迟 |
  | bge-large-zh 大模型 | 4 h | +1-3 pp | 易,显存需求增加 |
  | **Reranker 对比学习 LoRA** | 16-24 h | +5-8 pp | 难,需 1000+ 样本 |
- 为什么留给后续: 业界 RAG 做完这些也仅 0.45-0.55 的 Recall@5,是"下一阶段研究"而非"本项目缺陷"

## 段四 · 研究意义与贡献(0.5 分钟)

### 讲稿:

这个项目的核心学术贡献是: **RAG 仅能将幻觉砍一半,剩余部分必须靠指令微调和规则校验联合缓解**。

这在 **<1B 中文小模型**上给出的关键实证: 
- G0→G1 (纯 RAG): 幻觉 18% → 10% (-44%)  
- G1→G3 (加 LoRA): 幻觉 10% → 2.0% (-67%)  
- G0→G2 (超强 prompt): 幻觉仍然 10%,无改善(说明 <1B 模型 zero-shot 无法遵循复杂格式)

从部署角度,完整系统在 8GB GPU 上 42分钟完成训练,推理延迟仅增加 0.7 秒,成本可控。

我们还提供了:
- **30 条真实样本的 G0→G3 对比展示** (最有说服力)
- **8 条 YAML 校验规则**(可审计,易部署)
- **完整的数据构造 + 训练 + 评估流程**(可复现)

这套方案可以直接应用到任何垂直法律领域的问答系统。

## 快速参考 · 答辩中可能被问的问题

### Q1: "为什么微调后 EID 命中率只有 28%,这不是很低吗?"

A: G0/G1/G2 是 0%,G3 到 28% 已经是巨大进步。但你说得对,28% 确实还是太低,主要原因是**检索瓶颈** —— 我们的 Recall@5 仅 0.388(业界基准 0.45-0.55),即检索 top-5 有 61% 的概率根本找不到 gold case。这不是生成层 LoRA 的问题,而是"下一阶段研究",需要 16-24 小时的 Reranker 对比学习。

### Q2: "B1/B3 未修是不是偷懒?"

A: 不是。我们做的是**诚实的缺陷报告**。B1 修复成本 2-3 小时数据 + 30 min 重训,B3 需要 3-4 小时重训,但都**不改变核心结论** "RAG+LoRA 可将幻觉从 18% 降至 2.0%"。这是成熟项目的做法 —— 主动暴露边际收益小的 Bad Case,为下一步工作留出清晰路线图。

### Q3: "这个系统在实际部署中怎么保证不幻觉?"

A: 三层防线:
1. **检索层约束**: 提供真实证据,模型无法编造
2. **生成层约束**: LoRA 微调 + 强约束 prompt + 强制注入 top-1 EventID
3. **规则层约束**: L7 Validator 8 条 YAML 规则,任何超出证据的答案直接降级

实测幻觉率从 18% 降至 2.0%,部分来自"数字编造"(微调缓解),部分是"合理推理被认为是幻觉"(难以区分)。

### Q4: "性能 10+ 秒的延迟对实际应用可接受吗?"

A: 
- 系统现状: G3 延迟 10.54 秒
- 构成: 检索 3s + 排序 2s + LoRA 生成 4s + 冷启动等
- 改进空间: ①量化加速(目前 FP16 可降至 7s), ②Reranker 替换(可节省 1-2s), ③并行化(可节省 1s)
- 对标: 业界 RAG 系统通常 5-15s,我们在合理范围
- 合规场景: 大多数合规查询允许 5-10s 延迟(不是实时聊天)
