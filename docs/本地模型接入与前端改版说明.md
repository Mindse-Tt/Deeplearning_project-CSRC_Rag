# 本地模型接入与前端改版说明

## 这次改了什么

本轮把原来的“规则意图路由 + 模板回答 + 简单表单页”升级成了三段式系统：

1. 意图分类模型  
   位置：`src/csrc_rag/orchestration/intent_model.py`  
   方案：`TF-IDF(char_wb 2-4gram) + LogisticRegression`  
   作用：把问题分到 `case_retrieval / law_grounding / sanction_recommendation / trend_analysis`

2. 本地回复模型  
   位置：`src/csrc_rag/response/responder.py`  
   方案：默认接 `Qwen/Qwen2.5-0.5B-Instruct`，严格用检索证据构造 prompt；如果本地模型加载失败，则回退到模板生成

3. 聊天式前端  
   位置：`web/index.html`、`web/style.css`、`web/app.js`  
   参考：`LKWCoach` 的左侧会话栏、顶部状态栏、中间聊天区、底部大输入框布局  
   差异：本项目保留了证据卡片、意图置信度、过滤条件和命中案例展示，方便课程答辩

## 检索策略如何与本地模型配合

当前链路：

`意图分类 -> Query Plan -> Hybrid Retrieval -> 本地回复模型 -> 证据卡片展示`

其中：

- 意图分类模型决定问题类型和检索 top-k
- Query Plan 负责抽取年份、上市公司、监管机构等过滤条件
- RetrievalEngine 负责按事件级聚合和 chunk 召回
- 本地回复模型只读取召回到的案例证据，不直接自由生成

## 为什么这样设计

- 只靠规则路由不稳定，尤其是“处罚建议 + 法条依据”的混合问题容易误分
- 只靠模板回答不够像一个完成的 RAG 系统，也不利于展示“本地模型调用”
- 只做聊天界面不展示证据，答辩时很难证明系统不是幻觉

所以现在的设计目标是：

- 有模型化路由
- 有本地生成
- 有检索证据
- 有前端可视化

## 当前限制

- 本地回复模型第一次运行需要下载和加载权重，首轮响应会明显慢
- Dense retrieval 目前默认仍是 `svd_tfidf`，更适合稳定演示；后续可以切到真实中文 embedding
- 意图分类训练集目前是领域模板增强数据，不是人工标注真实用户查询集，论文里要如实披露
