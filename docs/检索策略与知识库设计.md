# 检索策略与知识库设计

这份文档回答四个问题：

1. 为什么知识库不只用 `M列 Activity`
2. 检索策略为什么这样定
3. 这个系统到底能解决什么问题
4. 为什么它比简单替代方案更稳

## 1. 知识库不只用 `Activity`

`Activity` 是主证据字段，但不是唯一字段。

如果只把 `M列 Activity` 直接扔进向量库，会有三个问题：

- 案件上下文不完整
- 法条依据容易缺失
- 相似案例即使行为相近，也可能因为机构、年份、处罚对象不同而误召回

因此当前知识库采用“主文本 + 辅助元数据 + 回答参考字段”的结构。

### 1.1 主文本

- `Activity`

作用：

- 作为主要语义检索内容
- 支撑案例相似性判断
- 提供问答证据片段

### 1.2 辅助字段

- `Law`
- `FileName`
- `Promulgator`
- `Supervisor`
- `DeclareDate`
- `ViolationType`
- `Party`
- `Position`
- `Relationship`

作用：

- 帮助系统缩小检索范围
- 提升回答可解释性
- 在问答阶段补充“法条、角色、监管机构、时间”的证据

### 1.3 回答参考字段

- `PunishmentType`
- `PunishmentMeasure`
- `SumPenalty`

作用：

- 用于在检索命中后生成推荐与展示结果
- 不能直接进入处罚预测模型输入，否则会泄漏标签

## 2. 知识库如何处理

当前知识库不是“整条记录直接入库”，而是分层处理。

### 2.1 事件级知识库

先按 `EventID` 聚合，得到事件级文档。

原因：

- 原始表混合了事件级与当事人级信息
- 不先聚合会导致同一事件重复入库，检索噪音过大

### 2.2 Chunk 策略

当前第一版 chunk 处理如下：

- 每个事件先生成一个 `summary chunk`
- 再对 `Activity` 做滑动窗口切块
- 对 `Law` 单独切块

当前默认参数：

- `chunk_size = 420`
- `chunk_overlap = 80`

这样做的原因：

- `Activity` 普遍较长，中位长度约 1500 字符，不能整段直接检索
- 如果块太短，语义不完整
- 如果块太长，BM25 和后续 reranker 都会被噪音拖累
- `Law` 与 `Activity` 的功能不同，单独切块更利于“法规依据问答”

### 2.3 当前生成的知识库层次

- `event_corpus.jsonl`：事件级主文档
- `event_chunks.jsonl`：可检索 chunk 库
- `party_samples.jsonl`：处罚类型预测训练样本

这三层分别对应：

- 检索
- 问答
- 监督学习

## 3. 检索策略为什么这样定

## 3.1 先做 BM25 baseline

当前第一版先落地 BM25，不是因为它最好，而是因为它最适合作为第一层可验证基线。

原因：

- 法律与监管文本中，固定术语、法规名称、主体名称很重要
- 在没有稳定 dense 模型和 reranker 之前，BM25 可以先验证知识库构造是否合理
- 课程项目需要清晰 baseline，对比链路不能一开始就过于复杂

## 3.2 为什么后面不是停在 BM25

BM25 的问题也很明显：

- 不能很好处理同义表达
- 对长句改写不够鲁棒
- 对“案情相似但措辞不同”的情况容易漏召回

因此后续检索策略升级路径已经定好：

1. `BM25`
2. `Dense Retrieval`
3. `Hybrid Retrieval`
4. `Hybrid + Reranker`

## 3.3 意图驱动的检索

系统不是所有问题都走同一条路。

当前已经定义四种意图：

- `case_retrieval`
- `law_grounding`
- `sanction_recommendation`
- `trend_analysis`

这样做的原因：

- “找相似案例”和“问法规依据”不是同一个检索目标
- “推荐处罚方式并说明依据”必须把处罚推荐作为主任务，而不是只找法条
- 后续前端接入后，不同页面本质上就是不同意图入口

## 3.4 元数据过滤

当前查询规划层已经支持从问题中抽取以下过滤条件：

- 年份
- 是否上市公司
- 监管机构提示

这一步的意义：

- 降低无关结果干扰
- 支持更细粒度问题
- 为后续 hybrid 和 reranker 提供更干净的候选集

## 4. 这个项目能解决什么问题

当前项目适合解决以下四类问题。

### 4.1 相似案例检索

例如：

- 这段违规行为最像哪些历史处罚案例？
- 某类行为过去通常怎么处罚？

### 4.2 法规依据问答

例如：

- 这类行为通常违反哪些法条？
- 相似案例里最常援引哪些规定？

### 4.3 处罚推荐

例如：

- 根据这段案情，更可能对应哪些处罚方式？
- 是否存在市场禁入风险？

### 4.4 趋势分析

例如：

- 近几年类似案件的处罚是否更严？
- 哪些监管机构对同类案件更常用罚款？

## 5. 为什么比简单替代方案更稳

这里的“更好”不能空说，必须和可比较的替代方案比。

### 5.1 比“只用 M 列直接搜”更好

我们的方案：

- 不只用 `Activity`
- 同时保留法规、机构、时间、主体等上下文
- 支持按意图控制不同回答结构

简单方案的问题：

- 很容易召回案情相似但监管背景不对的案例
- 回答里缺法条、缺机构、缺解释

### 5.2 比“整行直接入库”更好

我们的方案：

- 先做事件级聚合
- 再做 chunk 切分
- 分开知识库与预测样本

简单方案的问题：

- 同一事件重复很多次
- 当事人级与事件级粒度混乱
- 检索与训练容易互相污染

### 5.3 比“让大模型直接回答”更好

我们的方案：

- 先检索，再回答
- 强调证据与法规引用
- 可以量化检索效果与幻觉率

简单方案的问题：

- 容易幻觉
- 难以复现
- 很难做课程要求里的 baseline 和消融

### 5.4 比“只有分类器”更好

我们的方案：

- 既有处罚类型预测
- 又能给出相似案例与法规依据

简单方案的问题：

- 只能给标签，不能解释
- 不能做可交互问答
- 答辩展示价值弱

## 6. 当前版本的定位

当前已经落地的是第一版检索基线：

- 事件级知识库
- chunk 库
- BM25 检索
- 意图路由
- 查询规划
- 本地前端 Demo

这一版的作用不是“最终最强效果”，而是：

- 验证知识库设计是否合理
- 为后续 Dense / Hybrid / Reranker 提供对照组
- 支持前端和答辩的第一版演示

### 6.1 当前已验证结果

当前本地已经完成以下验证：

- `event_corpus.jsonl` 已生成，共 `4233` 个事件级文档
- `event_chunks.jsonl` 已生成，共 `29314` 个 chunk
- 本地前端 Demo 已能通过 `http://127.0.0.1:8000` 提供查询接口
- 真实查询“上市公司董事利用内幕信息买入股票获利，历史上通常对应哪些案例”能够召回明显相关的内幕交易案例

当前第一版自举检索 sanity benchmark 结果：

- `Recall@5 = 0.0933`
- `MRR = 0.0811`
- `nDCG@10 = 0.0822`

如何理解这组结果：

- 这说明第一版 BM25 baseline 已经不是空架子，能够工作并返回相关案例
- 但它离“高质量相似案例检索”还有明显差距
- 这正好为后续 Dense / Hybrid / Reranker 提供了清晰的提升空间

## 7. 后续升级路线

在当前 BM25 baseline 的基础上，下一步按这个顺序升级：

1. 加真实中文 Dense Embedding
2. 做 Hybrid 融合
3. 加 Reranker
4. 接 RAG 生成
5. 加处罚类型微调模型

这样升级的好处是每一步都有清晰对照，不会把系统复杂性一次性拉满。

### 7.1 当前 embedding 是怎么做的

当前已经接入了 dense retrieval 接口，但默认 backend 先选了 `svd_tfidf`，不是最终版的预训练 embedding 模型。

这样做的原因：

- 当前机器上可以立即跑通
- 可以先把 hybrid retrieval 的工程结构、指标和联调做起来
- 后续替换成 `sentence-transformer` 模型时，不需要重写上层检索编排

也就是说，当前状态是：

- BM25：已落地并验证
- Dense 接口：已落地并验证
- Dense 真实中文预训练模型：环境已准备好，后续只需替换 backend 和模型名

## 8. 当前训练路线

当前已经有两条训练路径：

### 8.1 可立即跑通的 baseline

- `TF-IDF + OneVsRest LogisticRegression`

这条路径已经完成本地训练验证，当前测试集结果为：

- `Micro-F1 = 0.8644`
- `Macro-F1 = 0.2759`
- `HammingLoss = 0.0691`
- `SubsetAccuracy = 0.5961`

如何理解：

- 说明 `PunishmentType` 的监督学习任务已经跑通
- 也说明当前标签高度不均衡，`Macro-F1` 仍然偏低

### 8.2 真正的 transformer 微调路径

当前已经完成：

- `.venv311` 环境搭建
- `torch / transformers / datasets / sentence-transformers` 安装
- `train_punishment_finetune.py` 脚本接入
- 基于 `hf-internal-testing/tiny-random-bert` 的小样本 smoke test 跑通

这说明：

- 训练型模型已经真正接进项目
- 后续把模型名切到 `hfl/chinese-macbert-base` 或其他中文模型即可进入正式微调实验

## 9. 文档结论

当前项目不是在做“普通搜索”，而是在做“面向监管处罚案例的证据驱动检索与回答系统”。

它的优势不在于某一个模型更大，而在于：

- 数据粒度处理更合理
- 检索对象设计更合理
- 支持意图路由
- 支持证据型回答
- 可以同时接检索、预测和前端演示
