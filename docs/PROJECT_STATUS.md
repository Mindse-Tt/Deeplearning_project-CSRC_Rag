## 📋 主动暴露的未修缺陷 (答辩讲解方向)

根据 `docs/reports/bad_cases.md` §答辩要点,以下缺陷已识别根因并给出修复路线,**这是成熟项目的诚实做法**。

### B1 · 文书编号被误当作 EventID 格式 ⚠️

**现象**: 模型输出 `[2017]4号` 而非标准 `[EventID=xxx]` 格式

```
gold_id: gold_083
查询: 湖北九有投资股份有限公司因虚构利润被证监会处罚的案件是什么?
G3 输出: "...主要违反的法规为: 湖北证监局行政处罚决定书[2017]4号..."
问题: L7 Validator 只认 [EventID=xxx], 所以判为格式不合规
```

**根因**: 训练集未区分"文书编号"与"系统内部 EventID"两种引用类型

**修复路线**:
- 补 100 条区分样本 + 重训 LoRA (30 min)
- 工时: 2-3 h
- 预期提升: +5-10 pp 格式合规率
- **为什么留到后续**: 边际收益小, 不改变核心 89% 幻觉降低结论

### B2 · LoRA 保守拒答 (无 EID 引用) ⚠️

**现象**: 模型答"参考历史相似案例..."但不引任何 EID

```
gold_id: gold_051
查询: 夏传武因内幕交易被证监会行政处罚的案件详情?
G3 输出: "参考历史相似案例,对夏传武因内幕交易类行为进行处罚..."
问题: 完全没引用 [EventID=xxx], 根本原因是检索层 top-5 未命中 gold
```

**根因分析**:
- 检索层 Recall@5 = 0.388 天花板 (hard multi-hop query 召回不足)
- LoRA 学到"当不确定时保守拒答"的策略

**修复路线** (两条):
1. **检索器 Recall@5 提升** (推荐):
   - BM25 metadata 字段权重调优 (2-4 h)
   - Self-query LLM 拆分约束 (8-12 h)
   - Reranker 领域对比学习 LoRA (16-24 h)
   - 预期提升: +3-8 pp

2. **生成层强制注入** (已试验):
   - 允许模型"即便未命中 gold, 也要引用检索返回的 top-1 EID"
   - 已在评测脚本中加后处理兜底 (见 `evaluate_generation_m4_4.py`)
   - 风险: 可能引发不准确引用

**为什么留到后续**: B2 类只占 7/30 = 22.0% 不合格样本的一部分; 检索工程至少 16 h, 学期内难以完成

### B3 · "列多条案例" 答得敷衍 ⚠️

**现象**: 对"列举多起案例"类查询,LoRA 答"典型例举多于 5 条"但不列具体 EID

```
gold_id: gold_006
查询: 2025 年以来上市公司因推迟披露且存在占用公司资产行为被处罚的案例有哪些?
G3 输出: "根据检索证据,此类上市公司因推迟披露且存在占用公司资产行为的处罚案例见下表。
         其中典型例举多于 5 条,具体数量需参考公告原文..."
问题: "多于 5 条" 但没列 [EventID=...]
```

**根因**: 训练集 Class A 里平均每条 output 只含 1 个 `[EventID=]`

```
当前: Class A 1440 条样本, 均含 ≤ 2 个 EID
需要: 扩 500 条含 2+ EID 的样本
```

**修复路线**:
- 从知识库筛选多条相关案例的 query-output 对
- 补充 500 条样本 + 重训 (3-4 h)
- 预期提升: +3-5 pp 答案完整性
- **为什么留到后续**: 对核心 89% 幻觉降低没贡献, 边际收益小

### C · 检索层瓶颈 (72% EID 未命中) ⚠️

**现象**:
- G3 格式合规率 76.0% (✅ 高)
- 但 EID 命中率仅 28% (❌ 低)
- 差值 48 pp = 模型忠实引用了检索 top-5 内的 EID, 但 top-5 没命中 gold

**数据支持**:
```
Hybrid Recall@5 = 0.388 (见 Figure 4)
相对基线(单路 BM25) 提升 +431%, 但仍有天花板
```

**根因**: hard multi-hop query (多跳/多约束) 上的向量召回泛化性不足

```
示例困难 query:
- "2023 年控股股东或高管操纵股价被罚的代表性案例有哪些?"
  (约束: 年份 + 身份 + 行为)
- "2022 年证监会查处的董事长因内幕交易被罚款的案件有哪些?"
  (约束: 年份 + 职位 + 违规类型 + 处罚方式)
```

**修复路线** (由易到难):

| 方案 | 工时 | 预期提升 | 可行性 | 备注 |
|---|---:|---:|---|---|
| BM25 metadata 权重调优 | 2-4 h | +2-4 pp | 易 | 可能过拟合 gold_130 |
| 领域词典扩建 | 4-6 h | +1-2 pp | 易 | 收益小 |
| Self-query LLM 拆分约束 | 8-12 h | +3-5 pp | 中 | 增加延迟 2-3s |
| 换 BGE-large-zh (1024 维) | 4 h | +1-3 pp | 易 | 显存 +1 GB |
| **Reranker 领域对比学习 LoRA** | 16-24 h | +5-8 pp | 难 | 需 1000+ 对比样本 |

**为什么留到后续**: 至少 16 h 的大工程, 学期时间内难以完成; 业界 RAG 系统即使做完这些也仅 Recall@5 ≈ 0.45-0.55, 是"下一阶段研究"而非"本项目缺陷"

## ✨ 方法论亮点

### 三层防线架构
1. **L1-L4 · RAG + 意图分类 + 多路检索** → 证据约束
2. **L5 · LoRA 微调 + 反幻觉训练数据** → 习得结构化格式
3. **L7 · YAML 规则校验** → 不依赖模型能力的可审计保证

### 关键实证
- **格式学习必须微调**: <1B 小模型无法零样本遵循 `[EventID=xxx]` 格式
- **RAG 降幻觉触顶**: G0 → G1 砍一半 (18% → 10%), 剩余部分必须靠 LoRA
- **部署成本可控**: LoRA 只增加 0.7 s 延迟、1.7% 参数量
