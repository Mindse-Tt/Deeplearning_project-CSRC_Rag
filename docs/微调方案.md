# 微调方案（Fine-tuning Plan）

> 本文档是赛道 B 的核心评分支撑，答辩老师最可能就这一页追问。
> 目标：说清"微调什么、怎么微调、微调前后如何对比"。

## 1. 为什么必须做微调

课程赛道 B 原文要求：

- 须展示**微调前后的效果对比**（定量指标 + 典型样例）
- 须识别并缓解大模型在特定领域下的**幻觉/错引法条**问题
- 须考虑实际部署负载（Token 消耗、推理延迟）与隐私合规

开题报告承诺："以开源中文基座大模型为底座，结合自构的证券合规问答对数据，采用**轻量级参数高效微调**方法进行领域适配。"

因此本项目必须完成一次 **PEFT (LoRA/QLoRA)** 微调，不能只做分类头微调。

## 2. 三条微调路径（主次分明）

| 编号 | 任务 | 模型 | 方法 | 数据 | 评分支撑 |
|-----|------|------|------|------|---------|
| ① **主线** | 生成式 RAG 问答 | `Qwen/Qwen2.5-1.5B-Instruct` | **QLoRA**（4-bit + LoRA rank=16） | 自构 5,000-8,000 条证券合规 QA 对 | "模型微调 10分" + "Baseline 对比 10分" |
| ② **辅线** | 处罚类型多标签分类 | `hfl/chinese-macbert-base` | 全参微调 | `party_samples.jsonl`（14,740 条） | 对照现有 TF-IDF baseline (Micro-F1=0.8644) |
| ③ 可选 | 检索 embedding 领域适配 | `BAAI/bge-small-zh-v1.5` | 对比学习（hard negative） | 事件级正负例对 | 提升 Recall@5 |

**优先级**：① > ② > ③。如时间紧，③ 可略，但 ① 和 ② 必须做出来。

## 3. 主线详细方案：Qwen2.5-1.5B + QLoRA

### 3.1 为什么选 Qwen2.5-1.5B 而不是 0.5B

| 指标 | 0.5B | 1.5B | 7B |
|------|------|------|-----|
| 中文指令遵循 | 偏弱 | 可用 | 优 |
| 本地 CPU 推理延迟（A6000/M1）| <1s | 2-4s | 10s+ |
| QLoRA 单卡显存（16GB）| 够 | 够 | 吃紧 |
| 证券合规答题质量 | 生硬模板 | 能组织论述 | 接近商用 |

- **1.5B** 是课程演示环境下效果/成本的甜点。
- 7B 虽强，但微调显存紧张，且本项目的训练数据量（~5k）带不动 7B 的全量微调收益。
- 备选：若显存只有 8GB，降级到 **Qwen2.5-0.5B + LoRA**，论文里老实标注资源限制。

### 3.2 训练数据构造（最关键）

目标：**5,000 ~ 8,000** 条高质量指令微调样本。

数据格式（Alpaca 风格 + system prompt）：

```jsonc
{
  "instruction": "根据以下证监会处罚案例的检索结果，回答用户的问题。必须明确引用来源（案例编号/法条），不得编造。",
  "input": "用户问题：上市公司实控人利用内幕信息买入自家股票，历史上通常怎么处罚？\n\n[检索到的证据]\n案例1（EventID=E_2021_0045）：...\n案例2（EventID=E_2019_0112）：...",
  "output": "根据检索到的历史案例，实控人利用内幕信息交易自家股票通常对应以下处罚：\n1) 警告 + 罚款，常见倍率为违法所得的 1-5 倍（参见案例 E_2021_0045）\n2) 视情节严重程度追加市场禁入（参见案例 E_2019_0112）\n适用法条主要为《证券法》第五十三条、第一百九十一条。\n\n注：以上结论基于检索证据生成，不构成法律意见。"
}
```

#### 数据来源分配（总量 ~5,360 条）

| 来源 | 数量 | 生成方式 |
|------|------|---------|
| A. 模板派生（案例检索） | 2,000 | 遍历 `event_corpus.jsonl`，用 Activity/Party/Position 字段填空生成"这类行为对应哪些案例"类问题 |
| B. 模板派生（法条依据） | 1,500 | 用 Law 字段生成"违反哪些法条"类问题 + 规范答案 |
| C. 模板派生（处罚推荐） | 1,500 | 用 Activity + PunishmentType + PunishmentMeasure 生成"推荐处罚方式"类问题 |
| D. 拒答负样本 | 300 | 写越界/闲聊问题，教模型说"本系统只回答证券违规相关问题" |
| E. API 蒸馏（后期补充） | 可选 200 | 用 GPT-4/DeepSeek 在少量案例上生成参考答案做数据增强 |

**注意**：构造答案必须强制插入"[引用：EventID=xxx]"和"[法条：《证券法》第xx条]"标记，这是让模型学"可溯源"的关键。

### 3.3 训练超参（QLoRA）

```python
model: Qwen/Qwen2.5-1.5B-Instruct
quantization: 4-bit NF4 (bitsandbytes)
lora_config:
  r: 16
  lora_alpha: 32
  lora_dropout: 0.05
  target_modules: [q_proj, k_proj, v_proj, o_proj, gate_proj, up_proj, down_proj]
training:
  batch_size: 2            # 本机 RTX 2060S 8GB；Colab T4 可用 4
  grad_accum: 8            # 本机 accum=8；Colab T4 用 4（有效 batch=16 保持一致）
  learning_rate: 2e-4
  num_epochs: 3
  max_seq_length: 2048
  warmup_ratio: 0.03
  lr_scheduler: cosine
  fp16: true               # Turing (RTX 2060S) 不支持 bf16，本机走 fp16；Colab T4 可改 bf16=true
```

- 预计单轮耗时：5000 样本 × 3 epoch ≈ **2-4 小时**（RTX 3060/4070 级别）
- 若本地无独立显卡，用 **Colab T4 / Kaggle P100** 训练，保存 LoRA 权重（仅几十 MB）拷回本地加载

### 3.4 输出集成到现有系统

训完后 `responder.py` 的 `local_hf` backend：

```python
# 加载流程
base_model = AutoModelForCausalLM.from_pretrained("Qwen/Qwen2.5-1.5B-Instruct", load_in_4bit=True)
model = PeftModel.from_pretrained(base_model, "artifacts/models/qwen_lora_csrc")
```

对 `configs/models.json` 新增字段：

```json
"response_generation": {
  "backend": "local_hf",
  "model_name": "Qwen/Qwen2.5-1.5B-Instruct",
  "lora_adapter": "artifacts/models/qwen_lora_csrc"
}
```

## 4. 辅线方案：MacBERT 分类微调

- 现有 `train_punishment_baseline.py` 跑的是 TF-IDF + LogReg，Micro-F1=0.8644 / Macro-F1=0.2759。
- 新跑 `train_punishment_finetune.py --model-name hfl/chinese-macbert-base`，完整 3 epoch。
- 预期效果：Micro-F1 小幅提升（0.87-0.89），但 Macro-F1 应显著提升（长尾标签更均衡）。
- 按 `EventID` 切分 + 时间切分，防止同事件泄漏。
- 成果写入论文"监督任务微调前后对比"章节。

## 5. 微调前后对比实验设计（论文 4.2 章）

### 5.1 生成模型对比（主线）

| 组号 | 配置 | 目的 |
|------|------|------|
| G0 | Qwen-1.5B 原始 + 无 RAG | 测基础模型幻觉 |
| G1 | Qwen-1.5B 原始 + RAG（BM25+Dense） | 测纯 RAG 贡献 |
| G2 | Qwen-1.5B 原始 + RAG + 强证据约束 prompt | 测 prompt engineering 贡献 |
| **G3** | **Qwen-1.5B LoRA + RAG + 强证据约束** | **本项目完整方案** |
| G4 | GPT-4 或 DeepSeek API + RAG | 上限对照（写"部署成本分析"章节时用） |

**评估集**：自构 50 条人工标注 QA（覆盖 4 类意图）。

**指标**：

- **ROUGE-L / BERTScore-F1**（答案与金标答案的相似度）
- **引用证据命中率**：生成答案提到的 EventID 是否在检索出的 top-k 里
- **法条正确率**：答案中引用的法条是否在检索证据中存在
- **幻觉率**：人工标注"是否编造不存在的案例/法条"
- **推理延迟** & **显存占用**

## 6. 数据口径校准（必须修）

⚠️ 开题报告写"CSMAR 14,743 行去重后 8,000 条"，但实际本地处理结果是：

- 14,740 行原始 → 4,233 个事件 / 14,740 条当事人样本

**论文中必须统一口径**：写作"CSMAR 14,740 条原始记录，按 `EventID` 聚合为 **4,233 个事件文档**（事件级知识库），并保留 **14,740 条当事人样本**（分类训练集）"。

## 7. 幻觉缓解的三层防线

答辩老师会追问："你怎么防止大模型瞎编？"

本项目的回答：

1. **系统边界层**（`topic_guard.py`）：硬关键词拦截越界问题
2. **意图路由层**：语义分类器判定越界 → 触发拒答
3. **Prompt 约束层**：强制要求"答案必须引用 EventID 和法条编号"
4. **微调对齐层**：LoRA 训练数据里 100% 样本都带引证，让模型学会这个风格
